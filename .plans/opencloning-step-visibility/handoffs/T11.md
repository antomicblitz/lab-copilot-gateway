# T11 Handoff — Expanded Writeback Package with Approval Hash Binding and Partial Failure Handling

**Slice 10** of the OpenCloning Step Visibility build plan.

## Summary

Expanded the OpenCloning writeback package to include the GenBank construct,
cloning history JSON, Markdown audit report, optional intermediate
attachments, a notebook entry appended to the experiment body, and metadata
provenance. One approval now binds to the exact package via a package hash.
Partial failures are reported clearly with upload IDs and rollback
instructions.

## Deliverables

### Modified: `src/lab_copilot_gateway/opencloning.py`

**New data structures:**

- `WritebackStepResult` — tracks each writeback step's name, status
  (`success`/`failed`/`skipped`), upload ID, and error message.
- `WritebackResult` — aggregate result with overall status
  (`success`/`partial`/`failed`), all upload IDs, notebook/metadata flags,
  step list, rollback instructions, and warnings.

**New module-level functions:**

- `_compute_writeback_package_hash()` — binds approval to the exact
  writeback package (GenBank, history JSON, audit report, notebook summary,
  selected intermediates, protocol refs, validation bundle version).
  Uses canonical-JSON + SHA-256 (same as `compute_args_hash`).
- `_wrap_amendment_html()` — wraps notebook HTML in a `<section>` marker,
  mirroring `ElabftwWriteAdapter._wrap_amendment` so the OpenCloning adapter
  can append notebook entries directly via the eLabFTW client.
- `_build_rollback_instructions()` — builds actionable rollback text listing
  uploaded attachment IDs to delete.
- `_step_flags()` — extracts success/failure flags from step results,
  keeping `_assemble_writeback_result` under complexity 15.

**New adapter methods:**

- `_compute_package_hash_for_writeback()` — renders reports + history JSON
  from strategy state, computes component hashes, returns the package hash.
- `_render_writeback_reports()` — renders notebook HTML + audit markdown
  (no upload). Extracted from `_generate_writeback_reports` (T10).
- `_build_history_json()` — builds history JSON bytes + filename. Extracted
  from `_execute_writeback` inline code.
- `_upload_genbank_step()` — uploads the GenBank construct.
- `_upload_history_step()` — uploads the history JSON (non-blocking).
- `_upload_audit_report_step()` — uploads the audit report (BLOCKING).
- `_upload_intermediates_step()` — uploads selected intermediate sequences
  (opt-in via `include_intermediate_attachments` checkbox).
- `_upload_selected_intermediates()` — iterates strategy sequences and
  uploads each selected one as a GenBank file.
- `_append_notebook_step()` — appends notebook summary to experiment body
  via `patch_experiment_body` with amendment wrapper.
- `_patch_metadata_step()` — patches metadata/provenance (non-blocking).
- `_assemble_writeback_result()` — determines overall status from step
  results using the partial failure rules.
- `_run_writeback_steps()` — orchestrates all steps in order, returns a
  `WritebackResult`.
- `_audit_writeback_error()` — audits a blocking step failure.
- `_audit_writeback_result()` — audits the final writeback result.
- `_build_writeback_opencloning_result()` — builds the `OpenCloningResult`
  from a `WritebackResult`.

**Modified `writeback_artifact()`:**

- Computes the package hash from actual content before consuming approval.
- If `approval_args["writeback_package_hash"]` is present, verifies it
  matches — fails closed (deny + raise) before consuming the approval token.
- Includes `writeback_package_hash` in the result dict.
- Rebuilds the validation bundle with draft checks set to "pass" based on

**Refactored `_execute_writeback()`:**

- Steps 1–6 (token, identity, policy, approval, client, GenBank rewrite)
  remain inline.
- Steps 7–9 (uploads, notebook append, metadata, audit) are delegated to
  `_run_writeback_steps()`, `_audit_writeback_result()`, and
  `_build_writeback_opencloning_result()`.
- The GenBank upload failure no longer raises `OpenCloningAdapterError` —
  it returns a `WritebackResult` with status "failed" and rollback
  instructions. This is the key behavior change from T10.

## Writeback Step Order

1. **Upload GenBank** (blocking — primary deliverable)
2. **Upload history JSON** (non-blocking — bonus reproducibility data)
3. **Upload audit report** (BLOCKING — audit trail must be complete)
4. **Upload intermediate attachments** (opt-in, non-blocking)
5. **Append notebook entry** (non-blocking, but affects status)
6. **Patch metadata/provenance** (non-blocking)

## Partial Failure Rules

| Step failure | Overall status | Rollback? | Notes |
| --- | --- | --- | --- |
| GenBank upload | `failed` | Yes | Primary deliverable missing |
| Audit report upload | `failed` | Yes | Audit trail incomplete |
| Notebook append | `partial` | Yes | Uploads succeeded but body not updated |
| Metadata patch | `success` + warning | No | Notebook is primary; metadata is best-effort |

## Approval Hash Binding

The package hash binds to:

- `artifact_manifest` (filename + size)
- `final_construct_hash` (SHA-256 of GenBank bytes)
- `history_json_hash` (SHA-256 of history JSON)
- `audit_report_hash` (SHA-256 of audit report markdown)
- `notebook_summary_hash` (SHA-256 of notebook summary HTML)
- `selected_intermediate_attachments` (list of selected IDs)
- `protocol_source_refs` (protocol references)
- `validation_bundle_version` (schema version string)

**Verification flow:**

1. Orchestrator computes the package hash before requesting approval.
2. Includes `writeback_package_hash` in `approval_args`.
3. `writeback_artifact` recomputes the hash from actual content.
4. If mismatch → fail closed (deny + raise) BEFORE consuming the approval.
5. If match (or field absent for backward compat) → proceed normally.

## Key Design Decisions

- **GenBank upload failure no longer raises**: The old behavior raised
  `OpenCloningAdapterError` on GenBank upload failure. The new behavior
  returns a `WritebackResult` with status "failed" and rollback
  instructions. This allows the caller to report what was attempted and
  what to clean up, rather than just propagating an exception.
- **Audit report upload is blocking**: Without the audit report, the
  writeback is incomplete — the audit trail has a gap. The writeback
  returns "failed" (not "partial") if the audit report upload fails.
- **Metadata patch is non-blocking**: The notebook entry is the primary
  deliverable. If the metadata patch fails, the writeback is still
  "success" with a warning — the user can see the notebook entry.
- **Backward compatibility**: If `writeback_package_hash` is absent from
  `approval_args`, the hash is computed and included in the result but
  not verified. This allows existing approval flows to work without
  changes.
- **Complexity ≤ 15**: All functions pass the cognitive complexity gate.
  The `_assemble_writeback_result` method was refactored to use
  `_step_flags()` to avoid exceeding the limit.

## Integration Points

- The `writeback_result` dict is available at
  `result.result["writeback_result"]` with `status`, `upload_ids`,
  `notebook_appended`, `metadata_patched`, `steps`, `rollback_instructions`,
  and `warnings`.
- The package hash is at `result.result["writeback_package_hash"]`.
- The plan executor checks `writeback_result.status` and marks the step as
  "failed" if the writeback failed or was partial.
- The `app.py` dispatch surfaces the validation bundle at the top level
  (unchanged from T9/T10).
