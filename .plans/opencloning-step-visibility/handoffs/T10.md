# T10 Handoff — Deterministic Notebook Summary and Markdown Audit Report Renderers

**Slice 9** of the OpenCloning Step Visibility build plan.

## Summary

Built two deterministic report renderers for the writeback flow. After a cloning
run completes, the gateway now generates:

1. A concise HTML notebook entry for the experiment body (via
   `amend_my_experiment_after_approval`).
2. A comprehensive Markdown audit report uploaded as a `.md` file attachment.

Both are deterministic — they use captured step/protocol/validation data as the
source of truth. The LLM may contribute rationale via the `llm_rationale`
parameter but cannot invent or override facts.

## Deliverables

### New file: `src/lab_copilot_gateway/opencloning_reports.py`

Two public renderers, each decomposed into many small helpers (all cognitive
complexity ≤ 15):

- **`render_notebook_summary()`** — Concise HTML for the experiment body:
  Objective, method, sequence/source counts, selected regions, primer table
  (with 30-nt truncated sequences), PCR product details, protocol conditions,
  final construct info, validation status (✅ Pass / ❌ Blocked), attachment
  IDs, and optional LLM rationale (clearly marked).

- **`render_cloning_audit_report_markdown()`** — Comprehensive Markdown
  attachment: Header (title, date, method, goal), full step timeline,
  intermediate products table, primer property table (name, sequence, length,
  Tm, GC%), protocol entries, fallback assumptions, full validation bundle,
  provenance source tree, warnings, attachment IDs, and rollback instructions.

Key design decisions:
- Missing facts render as `N/A` or `not checked` — never invented.
- `llm_rationale` is the ONLY LLM-contributed content, wrapped in an HTML
  `<details>` element with `AI-generated rationale` summary.
- No external dependencies — pure string concatenation/f-strings.
- Primer sequences > 30 nt are truncated with `…` in the notebook summary.

  renders both reports and uploads the Markdown audit report as an attachment.

  `_generate_writeback_reports()` as step 7c (after GenBank + history JSON

  with `notebook_draft_generated=True` and `audit_draft_generated=True` after
  successful report generation, so the bundle reflects the completed drafts.

## Key Design Decisions

- **Reports generated in writeback**: The reports are generated inside
  `_execute_writeback()` after the GenBank and history JSON uploads, so they
  can reference the upload IDs. The validation bundle is then rebuilt with
  draft checks set to "pass" in `writeback_artifact()`.
- **Non-fatal on errors**: If report generation or audit report upload fails,
  the writeback still succeeds — the artifact and history JSON are already
  uploaded. `notebook_summary_html` and `audit_report_markdown` will be `None`
  in the result dict.
- **Complexity ≤ 15**: All functions pass the cognitive complexity gate.
  Heavily-nested helpers (`_find_last_leaf_or_content`, `_notebook_pcr_details`,
  `_audit_provenance`) were extracted into smaller focused helpers.

## Integration Points

- The validation bundle's `notebook_summary_draft` and `audit_report_draft`
  checks now report `"pass"` instead of `"not_checked"` after reports are
  generated in writeback.
- The notebook summary HTML is available at
  `result.result["notebook_summary_html"]` for the amendment path (the outer
  layer can pass it to `amend_my_experiment_after_approval` as
  `amendment_html`).
- The Markdown audit report upload ID is available at
  `result.result["audit_report_upload_id"]`.
